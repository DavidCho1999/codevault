[1] 기본 개념
================================================================================

Agent(에이전트)란?
------------------
- 특정 작업을 독립적으로 수행하는 AI 어시스턴트
- 각자 고유한 컨텍스트 윈도우에서 실행
- 메인 대화의 컨텍스트를 오염시키지 않음

왜 사용하나?
------------
1. 컨텍스트 윈도우 관리 (토큰 절약)
2. 병렬 작업 가능 (3-5배 속도 향상)
3. 작업 전문화 (특정 역할에 집중)


================================================================================
[2] 내장 에이전트 (Built-in Agents)
================================================================================

Explore (탐색)
--------------
용도: 코드베이스 검색, 파일 발견, 분석
특징: 읽기 전용 (수정 불가)
도구: Glob, Grep, Read, 제한된 Bash
사용 예: "이 프로젝트 구조 파악해줘"

Plan (계획)
-----------
용도: 구현 계획 설계, 코드베이스 리서치
특징: Plan Mode에서 자동 사용
사용 예: "이 기능 어떻게 구현할지 계획해줘"

General-Purpose (범용)
----------------------
용도: 복잡한 멀티스텝 작업
특징: 모든 도구 접근 가능
사용 예: 탐색 + 수정이 모두 필요한 작업

Bash (명령어)
-------------
용도: 터미널 명령 실행
특징: git, npm, docker 등 실행

BORIS CHERNY (Claude Code 창시자)
---------------------------------
URL: https://venturebeat.com/technology/the-creator-of-claude-code-just-revealed-his-workflow-and-developers-are

[활용법]
- Agent를 "모듈화된 역할"로 취급
- 코딩을 파이프라인으로: spec -> draft -> simplify -> verify
- 각 단계마다 다른 "마인드" 활용

[실제 사용하는 Subagent]
1. code-simplifier: 코드 작성 후 단순화
2. verify-app: E2E 테스트 (브라우저 열어서 UI 검증)

[핵심 원칙]
"Claude에게 스스로 검증할 방법을 주면 품질이 2-3배 올라간다"

[워크플로우]
- 터미널 5개 + 웹 5-10개 = 병렬 세션 운영
- Plan Mode로 계획 합의 -> auto-accept로 1-shot 완성


SHRIVU SHANKAR (월 수십억 토큰 운영)
------------------------------------
URL: https://blog.sshh.io/p/how-i-use-every-claude-code-feature

[의견: 커스텀 서브에이전트의 문제점]
"이론상 최고의 기능이지만, 실제로는 두 가지 문제 발생"

문제 1: 컨텍스트 Gatekeeping
- PythonTests 에이전트 만들면 -> 테스트 컨텍스트가 메인에서 숨겨짐
- 메인 에이전트가 전체적으로 판단 불가

문제 2: 인간 워크플로우 강제
- Claude를 인간이 정의한 패턴에 가둠

[그의 해결책: Master-Clone 아키텍처]
- 커스텀 에이전트 대신 -> CLAUDE.md에 컨텍스트 제공
- 메인 에이전트가 Task(...) / Explore(...)로 알아서 위임
- "에이전트가 자기 자신의 복제본에게 위임하게 하라"

[결론]
"복잡한 커스텀 명령어 = 안티패턴"

================================================================================
[4] 커스텀 에이전트 만들기
================================================================================

위치
----
.claude/agents/ 폴더에 YAML 파일 생성

기본 구조
---------
---
name: verify-app
description: 앱 변경사항 테스트
tools: Bash, Read
model: inherit
---
프로젝트의 표준 명령으로 앱 변경사항을 검증하세요.
테스트 실행, UI 확인, 문제 발견 시 보고.

예시: code-simplifier
---------------------
---
name: code-simplifier
description: 코드 단순화
tools: Read, Edit
---
방금 작성된 코드를 검토하고 단순화하세요.
- 불필요한 복잡성 제거
- 가독성 향상
- 중복 제거


================================================================================
[5] 에이전트 실행 패턴
================================================================================

패턴 1: 순차 실행 (Sequential)
------------------------------
사용 시점: 작업 간 의존성이 있을 때
예시: DB 스키마 변경 -> API 수정 -> 프론트엔드 수정
결과: 정확한 결과

패턴 2: 병렬 실행 (Parallel)
----------------------------
사용 시점: 독립적인 작업들
예시: "4개 에이전트로 코드베이스 탐색해줘"
결과: 3-5배 속도 향상

프롬프트 예시:
"4개 태스크를 병렬로 실행해서 코드베이스 탐색해줘.
각 에이전트는 다른 디렉토리를 탐색해."

패턴 3: 파일 기반 위임 (File-based)
-----------------------------------
사용 시점: 컨텍스트 오염 방지
결과: 50-80% 토큰 절약

================================================================================
[8] 주의사항
================================================================================

1. 에이전트는 다른 에이전트 생성 불가
   - 중첩 위임 필요시 -> Skills 사용 또는 메인에서 체이닝

2. 토큰 비용 증가
   - 에이전트 체이닝 많으면 -> 사용량 급증
   - Max 플랜 한도 빨리 도달

3. 쓰기 작업 주의
   - 병렬 에이전트가 같은 파일 수정 -> 충돌 가능
   - 읽기 작업에 주로 사용 권장
