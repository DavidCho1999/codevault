1. BORIS CHERNY (Claude Code 창시자, Anthropic)
--------------------------------------------------------------------------------
URL: https://venturebeat.com/technology/the-creator-of-claude-code-just-revealed-his-workflow-and-developers-are

[사고방식]
- "병렬화가 복잡함을 이긴다" - 로컬 5개 + 웹 5~10개 세션 동시 운영
- "실수는 제도화된 지식이 된다" - CLAUDE.md에 실수 기록 → 팀 전체가 학습
- "계획 먼저, 실행은 나중" - Plan Mode로 합의될 때까지 반복, 그 후 auto-accept
- "Opus 4.5 + thinking = 느려도 결국 빠름" - 작은 모델은 수정에 더 많은 시간
- "좋은 계획이 정말 중요하다!" - 계획이 좋으면 Claude가 1-shot으로 완성

[핵심 워크플로우]
- 터미널 탭 1~5 번호 매기기, 시스템 알림으로 입력 필요 시점 확인
- /commit-push-pr 같은 slash command 하루 수십 번 사용
- PR에 @.claude 태그로 팀원들과 학습 공유


2. ADDY OSMANI (Google Chrome 14년, 저자)
--------------------------------------------------------------------------------
URL: https://addyosmani.com/blog/ai-coding-workflow/
책: "Beyond Vibe Coding" (O'Reilly)

[사고방식]
- "LLM은 assistant, 나는 senior dev" - AI가 날 대체하는 게 아니라 가속
- "AI-native = 위협이 아닌 파트너십 마인드셋"
- "AI가 날 대체할까?" → "AI가 이걸 더 빠르게/다르게 할 수 있을까?"로 질문 바꾸기
- "Vibe coding ≠ AI-assisted engineering" - 둘은 다름

[핵심 원칙]
- 작은 단계로 반복 개발 (한 번에 큰 결과물 요청 X)
- AI 코드는 "초안"으로 취급 - 반드시 이해 후 머지
- AI 탓 금지 - 커밋한 사람이 100% 책임
- 고속도로 위의 생산성 vs 호기심 마인드셋 주의

[PART 2: 공통된 핵심 사고방식]
================================================================================

1. AI는 곱셈이다 (덧셈 아님)
   → 내 실력 × AI = 결과
   → 기초가 0이면 AI 써도 0
   → 실력 있는 개발자일수록 더 큰 효과

2. 계획 없이 코딩하면 더 오래 걸린다
   → Plan Mode 스킵 = 수정에 2배 시간
   → "2분 계획 = 20분 리팩토링 절약" (Bairros)
   → "좋은 계획이 정말 중요하다!" (Boris)

3. 컨텍스트 관리 = 결과 품질
   → 많이 넣는다고 좋은 게 아님 (Ashley: 60% 규칙)
   → /clear 자주, /context로 모니터링
   → 세션 분리 = 각 작업에 집중

4. 실수를 시스템에 기록하라
   → CLAUDE.md에 기록 → 팀 전체가 학습
   → "실수가 규칙이 된다" (Boris)
   → 파일이 작동하니까 기억도 작동 (Zvi)

5. 병렬화로 생산성 극대화
   → 여러 세션 동시 운영
   → "병목은 생성이 아니라 리뷰" (Simon)
   → "10X = 1X + 10 Claude" (Elton)

6. AI는 assistant, 내가 senior dev
   → AI 코드 = 초안, 반드시 검토
   → AI 탓 금지, 커밋한 사람이 책임 (Addy)
   → "자신감 넘치는 주니어"로 취급 (Simon)

7. 검증 루프가 품질을 2~3배 올린다
   → AI에게 스스로 검증할 수단 제공
   → 테스트, bash, 브라우저 자동화 등
